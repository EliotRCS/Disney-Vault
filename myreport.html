Appraisal of Strengths & Weaknesses of Old and New Web Standards

Through the process of creating my website I used various features of Hypertext Markup Language (HTML) 5, a type of code that allows users to form the structure to their website. HTML 5 enabled me to easily navigate through my code to find specific areas I wanted to change, allowing me to fix errors or add new lines of code without having to look through every line. On the contrast, HTML 4 the previous version of code, did not have this semantic markup, instead have multiple ‘div’ elements, meaning when you have lots of code it takes longer to find what you’re specifically looking for as well as being unclear what each thing is or does. To emphasise this point over 40 different elements and attributes were added in HTML 5 from HTML 4, making it overall much clearer for people to read others lines of code and understand what they do. Furthermore, HTML 5 encompasses may features that required plugins in HTML 4, once more making web design clearer, especially those new to HTML because you no longer have research and use complex plugins. For instance Matthew B. Hoy wrote in his book on HTML 5 that this is “the most talked about new feature”. Replacing plugins like Flash in HTML 4, to be able to view videos or animations. But since the release of HTML 5 most browsers have stopped using plugins like these, following the growth in iphone and phone users that couldn’t handle these plugins. This means that when users visit websites they no longer will be met with errors if they don’t have the correct plugin or version. With HTML 5 it all comes freely when using modern browsers and websites not limiting any user interaction as well as allowing greater manipulation of the code that controls the videos or animations. Additionally, Simon Pieters, writing on behalf of W3C web standards also notes the ‘confusion’ that HTML 4 had with its layout. For example, Pieters writes that “the p element is itself a "block-level" element, but has a content model (layout) of "inline" [in HTML 4]”. This means that understanding how elements interact with each other was very unclear, made even worse when CSS also uses the same terms of ‘block-level’ and ‘inline’ in its own layout separate from HTML. However, with HTML 5 such ‘confusion’ is reduced by the removal of such terms from the layout rules. Though it should be noted that HTML 5 retained the HTML 4 feature to allow elements to be ‘part of none of them, one of them, or several of them’, demonstrating that HTML 4 was not completely redundant, instead it was fundamental in providing the base concepts and techniques to what we can do now with HTML 5. Other new features HTML 5 added that were useful in my website were, the API’s for even simple things like ‘alert()’ as an alternative to always having to go into the console to see ‘console.log’ if something has fired or not, resulting in error checks being made easier. Finally, though it may be shown that HTML 5 is without flaw, there is the inherent weakness of it being a new website development tool. Whilst HTML 4 has been in use for 10 years prior to HTML 5 release, as Hoy concluded “it may be several years before these features are universally supported”. In addition, more programmers understand HTML 4 which they’ve been using, but now must learn new keywords to be able to make modern sites which all takes time. This is made worse by HTML 5’s tendency to be modified and continually updated so developers keep having to update their websites.

In conclusion, HTML 5 has surpassed its predecessor HTML 4 through a combination of useful parts from HTML 4 as well as adding in multiple new features to overall boost the amount of options available to the developer, as I myself have seen and benefitted from, then passing such new abilities onto the users, creating websites that have never before been possible to create.

On My Website – Development and Techniques Used

As part of my websites validation check using W3C I found that the most common error was the incorrect practice of putting “” to close image tags being unaware that they are different from other tags which do need closing in that style, upon replacing the errors with the correct style I had removed ~30 of the errors. Following on, another ~40 errors were quickly removed once I realised to add “alt” tag to said images. This highlighted to me the importance of having “alt” tags as, though not crucial for the site to run sometimes, they are useful for keeping code clear and well ordered, thus part of the good practice I developed through the process. Continuing, the next error showed how some seemingly modern sites might still be using an older obsolete language, such as YouTube using “frameborder=0” in its embed videos when there’s a CSS equivalent now. Once I had uploaded my site to GitHub I was able to confirm that there were no validation errors left in my HTML.

The CSS through the W3C validator had only a few errors about formatting on ‘hover’, keeping background colours (in case images don’t load) simple and importantly to add semicolons into code. Though using the jeasyui did prompt many additional errors these were not things related to the code as well as other warnings being a result of the validator not recognising ‘-webkits-‘ rather than the code being incorrect. Taking this into account I could fix those issues leaving only the jeasyui ones that are unable to change. Using these validators ensures that our code is running and reminds us to use good web practices when coding to avoid issues.

Best Practice Techniques

It is important to keep files organised to be able to effectively navigate through the website when developing, for both finding and adding additional items. For instance, I kept my html files outside the other files and labelled them appropriately to know what each are and to function on GitHub. In the same way, labelling folders ‘Assets, CSS and JS’ meant images and addons as well as stylesheets and JS code were all easy to access for me and anyone looking at my work.

Secondly, markup structuring was achieved by maintain smooth and clear flow of tags connecting from parent to child elements throughout the development process. We make sure to have good structure since if there were to be an issue we can ascertain sooner what be causing it through seeing the association between tags, as opposed to a clump of text with no structure being unclear what ‘div’ or element is located where and affects what. See pictures section at the end for illustration.
Thirdly, though as outlined in the introduction HTML 5 is used far more widely on browsers, there is still a need for certain prefixes for some lines of code to be able to develop my sites interoperability, with the main one being ‘webkit’ for Safari to understand new techniques in CSS. Whilst successfully completing browser testing, see video guide on website for more, I was able to learn that different browsers while functioning display customised texts in different ways, reminding us of the other aspects that browsers still differ in beyond standard HTML.

My Website Features

One such key feature of my website I developed first was the unique method of navigating the site via hidden “mouseenter” events in JS moving the screen to an off-screen page. It was later when I developed this further with the “transform: translate” that things were harder. It wasn’t until I researched and found that I was miss-using “getElementById” that I otherwise manipulated the screen width successfully. Leading on I developed what became a large part of my JS, editing the iframe videos to stop, reset and play all through just clicking images. By doing so I also expanded my knowledge of jquery which allowed for greater manipulation of these videos. However, with this knowledge came my attempt to incorporate draggable themes into my sites customiser, but along with trying to take them outside of the list they started in, it proved too inefficient for my time and thereby I created my new idea to have the images clicked to show an image in a previously hidden space on the page. Incorporating local storage initially seemed like it was going to be challenging, until I thought of ways to easily store and then retrieve information I need from text. For instance, storing whether the images are “visible” in a variable which if true stores it in local storage, then retrieving it from storage by another ‘if its true’ do something, show image onload. Something different I found challenging was in CSS positioning the images, which weren’t helped by how my website was setup, with some ‘absolute top’ positions having to be 230% to appear on the right page. I was able to overcome this through logical try-and-error till the images appeared on the correct screen. On the contrary, I found some strong success with ‘keyframes’ animations which enabled me to create simple yet effective effects for my limited amount of text. Finally, early in my work I found the need to force my website to load not at the default top of the site, rather half way down the site, which is when I then learnt the “onbeforeunload” allowing me to create such a function to move the site to the real starting page.

In the future, the website could be developed further through incorporating other methods to replace “sticky” in older browsers that don’t yet have this feature, such as using ‘fixed’ for example to provide a more consistent viewing experience. Also looking into the different font rendering engines to allow me to set a standard text layout to provide the same site consistency. There is the option to add both music and a diverse range of films if the site were to be developed further, along with adding more responsive design for device beyond mobile and computers.
